## 1. 为什么需要 SKILL

在理解 SKILL 之前，先想一个场景：

你刚招了一位聪明的实习生，学历顶尖、学习能力极强。第一天他坐下来，你让他「帮我做一份竞品分析报告」。

他确实聪明，但他不知道：

- 你们公司的报告用什么格式
- 先做什么、再做什么（你们的标准流程）
- 哪些竞品最重要、从哪些维度分析
- 最终报告长什么样

于是他交上来一份「有自己发挥」的报告，和你预期差距很大，返工。

这就是没有 SKILL 的 AI 现状。**大语言模型（LLM）足够聪明，但「聪明」和「可靠地按你们团队的方式工作」是两件事。**

RAG 帮模型补充了「知道什么」（外部知识），Tool 给了模型「能做什么」（外部执行能力），而 **SKILL 解决的是「怎么做」**——把你们团队积累的最佳实践、标准流程、输出规范，固化成可复用的操作手册，让模型每次执行同类任务都稳定、可预期、符合团队标准。

## 2. SKILL 的本质：三个核心属性

### 属性一：程序性记忆（Procedural Memory）

SKILL 存储的是「怎么做」的知识，而不是「是什么」的知识。

- RAG 知识库：「苹果公司成立于 1976 年」→ 陈述性知识
- SKILL：「当需要写竞品分析时，先做市场规模研究，再做功能对比，最后做 SWOT 分析，输出格式为……」→ 程序性知识

就像骑自行车——你脑子里存的不是「自行车的百科知识」，而是「怎么保持平衡、怎么转弯」这类操作性知识。SKILL 存的就是 AI 版本的「操作性知识」。

### 属性二：渐进式披露（Progressive Disclosure）

SKILL **不是始终存在于上下文里的**，而是按需动态注入。

Agent 启动时只扫描每个 Skill 的极少量元数据（名字和描述），只占几百个 Token。当用户的请求命中某个 Skill 时，才把该 Skill 的完整内容加载进上下文窗口。

这解决了一个根本矛盾：技能越多越好，但上下文窗口有限。渐进式披露让你可以定义几十个 Skill，而不必担心把上下文撑爆。

### 属性三：可移植资产（Portable Asset）

**一次定义，跨工具复用。**

SKILL.md 是一个开放标准（Anthropic 于 2025 年 12 月正式发布，随后被 OpenAI、微软等跟进采纳）。同一个 Skill 文件，可以在 Claude Code、Cursor、OpenCode、OpenClaw、Hermes 等十六个以上支持该标准的工具里直接使用。

你的团队积累的 Skill 资产，不会因为换了工具而作废。

## 3. SKILL 文件的解剖

一个标准 Skill 是一个目录，核心是 `SKILL.md` 文件，可选附带脚本和参考资料。

### 目录结构

```
skill-name/
├── SKILL.md          # 核心（必须）
├── LICENSE.txt       # 授权说明（可选）
├── scripts/          # 可执行辅助脚本（可选）
└── references/       # 参考资料、模板、示例（可选）
```

### SKILL.md 的两个部分

```markdown
---
name: code-review
description: 当用户需要对代码进行 Review 时使用此技能。涵盖可读性、
             潜在 Bug、性能问题、安全漏洞等维度的检查，并按严重程度分级输出。
---

## 你的角色

你是一名资深后端工程师，专注于代码质量和可维护性。

## 执行步骤

1. 先通读全部代码，理解业务意图
2. 按以下维度检查：
   - 代码可读性（命名、注释）
   - 潜在 Bug 和边界条件
   - 性能问题（时间复杂度、数据库查询）
   - 安全漏洞（SQL 注入、XSS、越权）
3. 按「严重」「建议」「可选」三级分类输出

## 输出格式

[严重] 问题描述 → 修改建议（必须修复）
[建议] 问题描述 → 修改建议（强烈推荐）
[可选] 问题描述 → 修改建议（有空可做）
```

**两个部分的职责完全不同：**


| 部分                    | 内容                     | 加载时机      | Token 消耗                |
| --------------------- | ---------------------- | --------- | ----------------------- |
| YAML frontmatter（元数据） | `name` + `description` | 会话启动时始终加载 | 极少（~50 token）           |
| Markdown 正文（指令）       | 执行步骤、角色定义、输出格式         | 命中后才加载    | 按内容量（通常 500-5000 token） |


**description 是最关键的字段。** Agent 完全依赖它来判断「这个任务要不要用这个 Skill」。写得太模糊，Skill 无法被准确命中；写得过于宽泛，又可能误触发。好的 description 应该：

- 明确描述**触发场景**（「当用户需要……时」）
- 包含**关键词**（用户可能用到的表达方式）
- 说明**不适用场景**（避免误触发）

## 4. SKILL 的底层工作机制

### 4.1 扫描与花名册

Agent 启动时，会遍历所有预设的 Skill 目录，只读取每个 `SKILL.md` 的 frontmatter，建立一张「技能花名册」：

```json
[
  { "name": "code-review",      "description": "当需要对代码进行 Review 时..." },
  { "name": "frontend-design",  "description": "当用户要构建前端界面时..." },
  { "name": "pptx",             "description": "当需要创建演示文稿时..." },
  { "name": "weekly-report",    "description": "当需要生成周报时..." }
]
```

这张花名册很小，不会对上下文窗口造成压力。

### 4.2 语义匹配

用户消息到来时，LLM 对照花名册做**语义匹配**（不是关键词匹配）：

- 用户说「帮我做个幻灯片」→ 命中 `pptx` Skill
- 用户说「帮我做个 PPT 汇报」→ 同样命中 `pptx` Skill
- 用户说「帮我写一段 Python 代码」→ 不命中任何 Skill（凭通用能力作答）

语义匹配的范围仅限于 `description` 字段。这再次说明为什么 description 至关重要。

### 4.3 全文注入

命中之后，系统读取该 Skill 目录的完整内容，注入到当前上下文窗口：

```
上下文窗口（注入后）：

┌─────────────────────────────────────────────┐
│  系统提示 / Rules（始终存在）                  │
├─────────────────────────────────────────────┤
│  对话历史（之前的多轮交流）                    │
├─────────────────────────────────────────────┤
│  ← 刚注入：pptx/SKILL.md 完整内容             │
│     步骤指南 + scripts/ + references/         │
│     （本次任务才有，占 ~2K-8K token）           │
├─────────────────────────────────────────────┤
│  用户当前消息                                  │
└─────────────────────────────────────────────┘
```

注入后，LLM 就会按 Skill 里定义的步骤、角色、格式来执行任务，而不是凭直觉发挥。

### 4.4 脚本调用（可选）

如果 Skill 的 `scripts/` 目录里有可执行脚本，LLM 在执行过程中可以调用这些脚本来完成实际的文件操作。

例如，`pptx` Skill 的 `scripts/` 里有一个生成 `.pptx` 文件的 JavaScript 脚本，LLM 会在生成完幻灯片内容后，调用这个脚本把内容写入真实的文件，交付给用户可以直接打开的 `.pptx` 文件。

这是 Skill 从「给建议」升级为「真正动手做」的关键机制。

## 5. 一次会话里的完整生命周期

```
【阶段一：扫描】  ← 只发生一次，会话启动时

  Agent 初始化
    ↓
  遍历所有 Skill 目录
    ↓
  只读 YAML frontmatter（极少 token）
    ↓
  建立技能花名册，等待用户输入

─────────────────────────────────────────────

【阶段二：匹配】  ← 每次用户发消息时触发

  用户消息到达
    ↓
  LLM 对照花名册做语义匹配
    ↓
  ┌─────────────────┬──────────────────────┐
  │    未命中         │        命中           │
  └─────────────────┴──────────────────────┘
       ↓                      ↓
  不加载任何 Skill         进入阶段三

─────────────────────────────────────────────

【阶段三：注入】  ← 命中后，一次性完整加载

  读取完整 SKILL.md 正文
    +
  加载 references/ 参考资料
    +
  加载 scripts/ 脚本路径
    ↓
  全部注入上下文窗口

─────────────────────────────────────────────

【阶段四：执行】  ← Skill 内容驱动任务

  LLM 按 Skill 定义的步骤推进任务
    ↓
  必要时调用 scripts/ 中的脚本
    ↓
  输出结果

─────────────────────────────────────────────

【阶段五：保留或卸载】

  新消息到来？
    ↓
  ┌──────────────────────┬───────────────────┐
  │  同类任务，Skill 已在  │   会话结束         │
  │  上下文里，无需重加载  │                   │
  └──────────────────────┴───────────────────┘
                                 ↓
                         上下文清空，Skill 卸载
```

**一个需要注意的细节：** 一旦某个 Skill 被加载进上下文，它会一直「存活」到会话结束或被压缩摘要。如果会话很长，且先后用了多个不同的 Skill，上下文会不断累积。这时需要上下文压缩机制（把用不到的 Skill 内容摘要掉）来释放空间。

## 6. SKILL 与其他机制的边界

团队在使用 AI 工具时，经常搞混这几个概念。一张表说清楚：


| 机制                        | 加载时机        | 典型内容             | 解决的问题     |
| ------------------------- | ----------- | ---------------- | --------- |
| **System Prompt / Rules** | 始终存在于上下文    | 身份定义、全局约束、永远不做的事 | 模型是谁、底线在哪 |
| **RAG**                   | 用户提问时检索相关片段 | 知识库文档、内部资料、最新信息  | 补充「知道什么」  |
| **SKILL**                 | 命中时按需加载完整内容 | 操作手册、执行步骤、输出格式   | 固化「怎么做」   |
| **Tool / MCP**            | 执行时调用外部系统   | API 接口、数据库、文件操作  | 扩展「能做什么」  |


**一个典型的混淆场景：**

- ❌ 错误做法：把「代码 Review 的步骤」写进 System Prompt
  - 后果：无论用户说什么，这段内容都占着上下文空间
- ✅ 正确做法：把它封装成 `code-review` Skill
  - 效果：只有用户明确需要 Code Review 时才加载，其他时候不占空间

**判断该用哪个的简单原则：**

- 这是对所有任务都有效的约束吗？→ Rules
- 这是需要实时查阅的外部知识吗？→ RAG
- 这是某类任务的标准执行流程吗？→ SKILL
- 这需要真的执行某个操作（发邮件、查数据库）吗？→ Tool / MCP

## 7. 六大工具的 SKILL 实践

### 7.1 Claude Code：规范的起源与原型

Claude Code 是 Anthropic 官方的命令行 Agent 工具，也是 SKILL.md 这套开放规范的起源地（Anthropic 于 2025 年 12 月正式发布）。

**Skill 发现路径（优先级从高到低）：**

```
项目级 Skill（随代码库走，可提交 Git）
  .claude/skills/<skill-name>/SKILL.md

全局 Skill（跨项目通用）
  ~/.claude/skills/<skill-name>/SKILL.md

内置 Skill（Anthropic 官方维护）
  随 Claude Code 本体安装
```

**核心特点：**

- 同时定义了 `CLAUDE.md`（项目规则文件）和 Skill 体系的分层，是整个行业的设计参考
- 官方维护的 Skill 生态最完善，涵盖前端开发、文档生成、PDF 处理、PPT 制作等
- Skill 可以附带可执行脚本，实现真正的文件生成（而不仅仅是给出文字建议）

**适合场景：** 工程化的 Skill 资产管理，大团队的标准化 AI 工作流

---

### 7.2 Cursor：Rules + SKILL 的双轨融合

Cursor 是最广泛使用的 AI 代码编辑器，其上下文定制体系历经多次演进，现在形成了 Rules 和 Skill 的双轨体系。

**第一轨：Rules（规则层）**——始终存在的约束

Cursor 的 Rules 分四种类型：


| 类型              | 触发方式          | 适用场景                     |
| --------------- | ------------- | ------------------------ |
| Always          | 始终生效          | 全局风格约束、技术栈声明             |
| Auto-Attached   | 匹配文件类型时自动加载   | TypeScript 规范（匹配 `*.ts`） |
| Agent-Requested | Agent 判断需要时加载 | 按需的参考文档                  |
| Manual          | 用户手动触发        | 不常用但需要时可调的规则             |


**第二轨：SKILL（技能层）**——动态加载的操作手册

Cursor 的 Skill 最强大的特性是可以携带 **Hooks**——在 Agent 执行某步骤之前或之后自动触发的脚本：

```javascript
// 示例：让 Agent 持续运行直到所有测试通过
const input = await Bun.stdin.json();
const scratchpad = existsSync(".cursor/scratchpad.md")
  ? readFileSync(".cursor/scratchpad.md", "utf-8") : "";

if (scratchpad.includes("DONE")) {
  // 任务完成，允许 Agent 停止
  console.log(JSON.stringify({}));
} else {
  // 未完成，推送继续执行的消息
  console.log(JSON.stringify({
    followup_message: `继续工作，完成后在 scratchpad.md 写入 DONE`
  }));
}
```

Hooks 把 Skill 从「静态说明书」升级成了「带反馈循环的自动化流程」。

**适合场景：** IDE 内长时间运行的自动化任务，需要「做完才停」的持续型 Agent 任务

### 7.3 Qoder（阿里巴巴）：Skill 融入 Spec 驱动开发

Qoder 是阿里巴巴于 2025 年 8 月发布的 Agent 编程平台，将 Skill 的理念深度融入了「**Spec 驱动开发**」工作流。

**两种模式与 Skill 的关系：**

```
Agent 模式（轻量 Skill）
  类似传统「对话 + 技能加载」
  Agent 根据请求上下文判断加载哪些技能

Quest 模式（Skill 即 Spec）
  开发者描述需求
    ↓
  Qoder 生成技术 Spec（这份文档充当任务专属 Skill）
    ↓
  Agent 执行完整的规划→编码→测试→报告全流程
```

**Qoder 的独特之处：Repo Wiki**

Qoder 会自动分析整个代码库的架构、模块关系、设计模式，生成并维护一份 `.qoder/repowiki/` 文档——相当于自动生成一个永久性的、始终最新的「项目 Skill」，团队所有人提交 Git 后共享。

**适合场景：** 大型功能的完整端到端开发委托，需要深度理解代码库架构的复杂任务

### 7.4 OpenClaw：最活跃的社区 Skill 生态

OpenClaw（前身 Clawdbot）是奥地利开发者 Peter Steinberger 于 2025 年 11 月发布的开源个人 AI Agent，两个月内成为 GitHub 历史上增长最快的开源项目之一（超过 345,000 stars）。

**四层配置文件体系：**

```
AGENTS.md   → 定义 Agent 的身份和通用行为
SOUL.md     → 定义 Agent 的「人格」和价值观
TOOLS.md    → 定义可用工具的约定
skills/
  └── <skill-name>/
        └── SKILL.md  ← 具体能力模块
```

**Skill 的发现优先级：**

```
工作区级别（最高优先级）
  ~/.openclaw/workspace/skills/<skill>/SKILL.md

全局安装（次优先）
  ~/.openclaw/skills/<skill>/SKILL.md

内置捆绑（最低优先）
  随 OpenClaw 本体安装
```

**Agent 执行循环：**

当消息到达时，OpenClaw 会组装上下文（对话历史 + 工作区文件），发送给 LLM，接收响应，执行工具调用，返回结果。如果需要多个工具才能完成任务，这个循环每次请求最多重复 20 次。

OpenClaw 的 **ClawHub** 是目前最活跃的公开 Skill 社区，包含数千个社区构建的技能。

⚠️ **安全警告：** OpenClaw 因其极高的系统访问权限而存在安全风险。建议像对待代码一样对待第三方 Skill——在安装前仔细阅读内容，只从可信来源安装。

**适合场景：** 个人自动化工作流，消息平台（WhatsApp、Telegram、Slack）集成，需要 24/7 后台运行的 Agent

---

### 7.5 Hermes Agent（Nous Research）：能自我进化的 Skill

Hermes Agent 是 Nous Research 于 2026 年 2 月发布的开源自主 AI Agent，解决了其他所有工具都没有解决的问题：**让 Agent 自己创建和改进 Skill**。

**自我进化的飞轮：**

```
用户给 Agent 一个复杂任务（≥5 个工具调用）
         ↓
Agent 完成任务
         ↓
Agent 分析本次执行过程
         ↓
自动生成 SKILL.md，记录：
  - 这类任务的标准流程
  - 已知的坑和边界条件
  - 验证步骤
         ↓
下次遇到相似任务，直接调用这个 Skill
         ↓
Agent 越用越强，不断积累专属技能库
```

**三层知识体系协同：**


| 层次    | 文件                  | 内容              | 大小限制      |
| ----- | ------------------- | --------------- | --------- |
| 技能层   | `skills/*/SKILL.md` | 操作手册（自动生成或手动创建） | 无固定限制     |
| 事实记忆层 | `MEMORY.md`         | 学到的项目事实、技术决策    | ~2,200 字符 |
| 用户理解层 | `USER.md`           | 你的工作风格、偏好、习惯    | ~1,375 字符 |


内存大小被刻意限制，以防止膨胀：小到不会让上下文窗口膨胀，但足以容纳 15-20 条有用的条目。

**适合场景：** 需要持续学习的长期个人 Agent，随时间积累越来越「懂你」的工作助手

### 7.6 OpenCode：互操作性最好的 Skill 兼容层

OpenCode 是一个开源的终端 AI Agent，其最大特点是**对所有主流配置格式的完整兼容**——你不需要迁移任何已有的 Skill 资产。

**Skill 发现路径（最完整的兼容层）：**

```
项目级（从当前目录向上递归到 Git 根目录）：
  .opencode/skills/*/SKILL.md        ← OpenCode 原生
  .claude/skills/*/SKILL.md          ← Claude Code 兼容
  .agents/skills/*/SKILL.md          ← 通用兼容

全局：
  ~/.config/opencode/skills/*/SKILL.md   ← OpenCode 原生
  ~/.claude/skills/*/SKILL.md            ← Claude Code 兼容
  ~/.agents/skills/*/SKILL.md            ← 通用兼容
```

**Rules 文件也兼容多种格式：**

```
优先级（第一个命中的生效）：
  AGENTS.md > CLAUDE.md（Claude Code 格式）> 其他
```

这意味着：你的 Claude Code Skill 无需任何修改就可以在 OpenCode 里直接使用。

**OpenCode 的 Agent 体系：**

OpenCode 不仅支持 Skill，还支持定义专用 Agent（用 Markdown 文件配置），每个 Agent 可以指定模型、温度、工具权限：

```markdown
---
id: coding-review
name: Coding Review
mode: primary
temperature: 0.1
tools:
  read: true
  edit: false
  write: false
---

# Code Review Agent
你是一名专注于代码质量的资深工程师...
```

**适合场景：** 多工具混用的团队（不同成员用不同工具），跨 Agent 迁移，希望一套 Skill 资产通用的团队

---

### 六大工具横向对比


| 工具              | 核心配置文件                   | 独特设计                 | 最适合场景              |
| --------------- | ------------------------ | -------------------- | ------------------ |
| **Claude Code** | `SKILL.md`, `CLAUDE.md`  | 官方规范起源，内置 Skill 库最完整 | 工程化 Skill 资产管理，大团队 |
| **Cursor**      | `SKILL.md`, `.mdc Rules` | Hooks 机制，四种 Rule 类型  | IDE 内长时间运行的自动化任务   |
| **Qoder**       | Spec 文档, Repo Wiki       | Spec 即 Skill，自动知识图谱  | 大型功能端到端开发委托        |
| **OpenClaw**    | `SKILL.md`, `SOUL.md`    | SOUL.md 人格定义，超大社区生态  | 个人自动化，消息平台集成       |

| **OpenCode**    | `SKILL.md`, `AGENTS.md`  | 兼容所有格式，最强互操作性        | 多工具混用团队，跨工具迁移      |

## 8. 如何写好一个 SKILL

### 8.1 确定是否值得封装成 Skill

在写 Skill 之前，先问三个问题：

1. **这个任务会重复发生吗？** 一次性的任务不值得封装。
2. **执行过程有没有「讲究」？** 如果随便做也差不多，Skill 价值不大。
3. **不同人做出来差距大吗？** 差距大说明有标准化的空间。

符合「重复 + 有讲究 + 差距大」三个条件的任务，才是 Skill 的最佳候选。

### 8.2 Skill 的结构设计

**description 的写法（最重要）：**

```markdown
---
name: prd-writing
description: 当用户需要撰写产品需求文档（PRD）、功能说明书、需求规格书时使用。
             适用场景：新功能开发前的需求整理、产品迭代文档、技术评审前的说明书。
             不适用：项目总结报告、用户调研报告（使用其他 Skill）。
---
```

好的 description 包含：触发场景 + 关键词 + 不适用说明

**正文的结构建议：**

```markdown
## 背景与角色
（告诉模型：你是谁，这个任务的背景是什么）

## 执行步骤
（核心：先做什么，再做什么，怎么做）

## 输出格式
（期望的结果长什么样，可以附模板或示例）

## 注意事项
（常见的坑，需要特别关注的点）

## 示例
（可选，效果最好。给一个「输入→输出」的完整示例）
```

### 8.3 一个完整的示例

```markdown
---
name: weekly-report
description: 当用户需要生成、撰写、整理周报时使用此技能。
             可接受原始工作记录（如 commit 记录、Jira 任务列表、随手笔记）作为输入，
             输出结构化的周报。不适用于月报、季报（使用 monthly-report Skill）。
---

## 背景与角色

你是用户的工作助手，负责帮助将一周的工作记录整理成结构清晰的周报。
周报的读者是直属上级，关注：完成了什么、遇到了什么问题、下周计划。

## 执行步骤

1. 仔细阅读用户提供的原始工作记录
2. 按「已完成工作」「遇到的问题与解决方案」「下周计划」三个板块分类整理
3. 每个板块内按重要性排序（最重要的放最前）
4. 语言简洁，避免技术术语（除非读者是技术背景）
5. 总长度控制在 300-500 字

## 输出格式

**本周工作汇报（YYYY.MM.DD - YYYY.MM.DD）**

**已完成**
- [重要程度 ★★★] 事项描述，结果说明
- [重要程度 ★★☆] 事项描述，结果说明

**遇到的问题**
- 问题描述 → 解决方案（或：尚未解决，计划……）

**下周计划**
- 优先事项一
- 优先事项二

## 注意事项

- 不要把技术细节罗列上去，要翻译成「做了什么、为什么重要」
- 如果本周什么都没完成，需要如实说明原因
- 请求确认：如果原始记录不够清晰，先向用户确认再输出
```

### 8.4 团队 Skill 库的维护建议


| 建议          | 说明                                            |
| ----------- | --------------------------------------------- |
| 用 Git 版本管理  | Skill 是团队资产，提交到代码仓库，走 Code Review             |
| 建立命名规范      | 按功能域分类：`dev-*`（开发类）、`doc-*`（文档类）、`ops-*`（运维类） |
| 定期复盘和更新     | 团队的最佳实践会演进，Skill 也需要跟着迭代                      |
| 新人入职即可用     | Skill 库本身就是最好的知识传承载体                          |
| 避免 Skill 膨胀 | 定期清理很少被触发的 Skill，质量 > 数量                      |

### 主流 Skill 社区

- **skills.sh**（Vercel 主导）：官方目录，按真实安装量排名，可信度高

- **awesome-hermes-agent**：Hermes 专用技能社区

### 安全提示

SKILL.md 的格式开放、门槛低——这既是生态爆发的原因，也带来了安全风险。2026 年初，Snyk 对近 4000 个公开 Skill 的审计显示，超过 36% 包含至少一个安全问题。

**核心原则：像对待 npm 包一样对待 Skill——安装前读内容，只信任可信来源。**

## 10. 常见问题

**Q：Skill 和 System Prompt 有什么区别？**

System Prompt 始终占用上下文空间，无论当前任务是否相关。Skill 是按需加载的，只有在任务命中时才会出现在上下文里。对于团队来说，应该把「全局规则」放 System Prompt / Rules，把「特定任务的操作手册」封装成 Skill。

**Q：一个 Skill 的 SKILL.md 应该写多长？**

没有硬性上限，但要考虑 token 消耗。通常 500-3000 字是比较合适的范围。太短不够用，太长既浪费 token，也可能引入干扰。如果指令特别多，可以把详细参考资料放到 `references/` 目录里，SKILL.md 正文只保留核心步骤。

**Q：如果多个 Skill 同时被命中怎么办？**

这取决于具体工具的实现。通常的处理方式是：命中置信度最高的那个优先全量加载，其他的可能只加载 description 作为参考。在设计 Skill 时，description 写得越精准，就越不容易产生混淆。

**Q：Skill 里可以引用另一个 Skill 吗？**

目前大多数工具不原生支持 Skill 嵌套调用。但你可以在 SKILL.md 正文里明确写「如果需要生成最终的 Word 文件，请调用 docx Skill」，LLM 会理解这个指引并在适当时机触发。

**Q：怎么判断一个 Skill 写得好不好？**

最简单的标准：**把 SKILL.md 给一个不了解这项工作的新同事读，他能按照步骤独立完成任务吗？** 能，说明 Skill 写得足够清晰。另外，一个好 Skill 应该让 AI 的输出和「团队里最资深的人的输出」无限接近。
