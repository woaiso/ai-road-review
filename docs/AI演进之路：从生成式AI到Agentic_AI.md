# 从生成式 AI 到 Agentic AI：一场正在发生的范式革命

> **写给所有人的 AI 技术全景导览**
> 不需要数学背景，只需要好奇心。

---

## 写在最前面

你一定打过这样的客服电话：

接通后，机器人说——「您好，欢迎致电XX银行。查询余额请按1，转账汇款请按2，信用卡业务请按3，更多服务请按4……」你按了4，它继续说「积分兑换请按1，账单分期请按2……」你按错了，它说「您的输入有误，请重新选择」，然后把你踢回到最开始。

这就是第一代「智能客服」。本质上，它是一棵**决策树**——程序员把所有可能的问题和答案提前写死，用户只能顺着树枝往下走，没有任何理解能力。

后来，有了能聊天的客服机器人。你打字说「我想查一下上个月的账单」，它能认出「账单」这个关键词，返回账单查询的链接。进步了，但还是机械的——换个说法「帮我看看上个月花了多少钱」，它可能就懵了。

今天，换成大语言模型驱动的 AI 客服，不管你说「账单」「消费记录」「上个月花了啥」还是「钱去哪了」，它都能理解你的意图，流畅作答，还能结合你的消费习惯给出理财建议。

而未来的 **AI Agent**，你根本不需要打电话——它主动发现你有一笔即将到期的定存，自动对比了利率，帮你预约了转存，起草了通知短信等你确认，全程无需你操心任何一个步骤。

从「按键迷宫」到「听懂你说话」到「主动替你办事」，这三级跨越，就是这篇文章要讲的故事。

# 第一章：破冰——我们在哪里，要去哪里

## 1.1 一张图看懂整个演进脉络

整个 AI 技术的演进，可以用一栋「智能大厦」来比喻。和常见的“技术名词堆叠图”不同，这张图强调的是**能力如何一层层长出来**：

```
┌──────────────────────────────────────────────────────────┐
│ L10-L11：范式与愿景层（Agentic AI + AGI）                │
│  - Agentic AI：为了目标自主完成                           │
│  - AGI：更长期的愿景                                     │
├──────────────────────────────────────────────────────────┤
│ L9：工程层（Harness）                                     │
│  - 把实验能力变成生产能力：可靠运行、可观测、可治理        │
├──────────────────────────────────────────────────────────┤
│ L5-L8：知识执行层（RAG + Tool + Skill / MCP + Agent）     │
│  - 按历史演进：先工具调用，再能力封装与协议化接入          │
├──────────────────────────────────────────────────────────┤
│ L3-L4：认知层（Thinking + Context）                       │
│  - 让模型会思考、会利用上下文                             │
├──────────────────────────────────────────────────────────┤
│ L1-L2：地基层（GenAI + Prompt）                           │
│  - 先让模型会说人话                                       │
└──────────────────────────────────────────────────────────┘
```

你可以用一句话记住这张图：**Harness 解决的是“如何可靠地跑”，Agentic AI 解决的是“为了目标自主完成”，AGI 代表的是更长期的愿景。**

再补一个底层视角：在这栋大厦的最底部，仍是数值计算链路（Embedding -> Attention -> FFN -> LM Head）；从下到上的每一层，本质上都是一次能力跃迁，让 AI 从“会写”逐步走向“会办”。

**伏笔**：图你已经看懂了，但“为什么偏偏是现在发生跃迁”还没回答。下一节，我们把时间轴拉开，看拐点是怎么一步步形成的。

## 1.2 为什么说现在是「范式跃迁」

2010 年代初，深度学习让 AI 在感知任务上突破（如图像分类）。  
2017 年前后，Transformer 奠定了现代大模型的核心架构。  
2022 年起，对话式 LLM 让普通用户开始高频使用 AI。  
2024 年后，Agent 工程化能力快速成熟并进入加速落地期。
2026 年起，Agentic AI 开始进入规模化落地阶段。

这不是量变，而是质变。就像从马车到汽车，并不是「快一点的马车」，而是**整个出行方式的重构**。

业界给这次跃迁起了个名字——**从 Generative AI（生成式 AI）到 Agentic AI（代理式 AI）**。前者关心「生成什么」，后者关心「达成什么」。
这里也要区分两个层级：**Agentic AI 是范式目标（为了目标自主完成），AI Agent 是具体实现形态（在某个系统里多步决策、调用能力并闭环执行）**。

在工程里，「会办事」最常用、也最好讲清楚的落点，是 **AI Agent**——第九章就专门讲这个：**多步决策、调工具、跑闭环**；宏观范式词汇不必在这一章里再铺一层。

既然“会办事”建立在“会生成”之上，那我们得先把底层搞清楚：LLM 到底是什么、它为什么能生成。第二章从这里开始拆。

# 第二章：地基——生成式 AI 的底层逻辑

## 2.1 LLM 是什么：一个会“续写下一句”的概率引擎

先给一句最短定义：

**LLM（Large Language Model，大语言模型）= 在海量文本上训练出来的“下一个 Token 预测器”。**

它并不是在脑子里存了一本百科全书，然后“查到答案再读给你听”；它的核心动作是：

1. 读取你给的上下文（问题、指令、历史对话）
2. 预测“下一个最可能出现的 Token”
3. 把这个 Token 接到上下文后面
4. 再预测下一个 Token
5. 周而复始，直到形成完整回答

你看到的一段自然语言，本质上是这个“连续预测-连续生成”的过程。

如果用一个更接地气的比喻：**它的本质有点像在做“成语接龙”**——前一个词会约束后一个词，后一个词又会继续约束下一步；模型就是在这种“基于上文接下文”的链式过程中，逐步把整段答案接出来。

> 🌟 **类比**：像手机输入法的“下一词联想”，但规模扩大了很多个数量级。输入法能补全一个词，LLM 能补全一段解释、一篇文章，甚至一份可运行代码。

### LLM 为什么看起来“像理解了你”

因为它在训练中见过海量语言模式，学会了：

- 词与词如何组合；
- 句子在不同语境下如何衔接；
- 什么表达通常更符合人类习惯。

所以它不只是复述原句，而是能在新语境下重新组织表达，给出看似“理解后”的回答。

### 一句话区分 LLM 与传统规则系统

- 规则系统：按预设分支匹配，命中就返回，不命中就失败。
- LLM：基于概率连续生成，即便没见过完全同一句提问，也能组合出合理回答。

这也是为什么后面我们要继续讲 `Token`、`Embedding`、`Self-Attention` 和训练过程：它们共同决定了 LLM 的“续写能力”是怎么形成的。

### 开源大模型到底“开源”了什么：本质是权重文件

很多人以为“开源大模型”主要是开源代码。其实在工程实践里，更核心的是：**开源了权重文件（Weights）**。

- 推理代码（Transformer 框架）很多团队都能写，结构也相对标准化；
- 真正昂贵、稀缺、决定能力上限的，是训练后得到的权重参数；
- 所谓“发布一个 7B / 70B 模型”，本质就是发布了一组可加载的权重文件。

那权重文件里“本质的东西”是什么？

你可以把它理解为：**模型在海量语料上学到的统计规律，被压缩进了几十亿到上千亿个数字里。**

这些数字不是可直接阅读的知识条目（它不会在文件里写着“法国首都是巴黎”），而是分布在不同层中的参数：

- 哪些词在什么语境下更可能一起出现；
- 哪类句式在什么任务下更合理；
- 语义之间在向量空间中的相对关系；
- 注意力该优先看哪里、忽略哪里。

所以，权重文件更像是“语言规律的参数化记忆”，而不是“人类可读的百科词条”。

> 🌟 **类比**：如果代码是“乐谱规则”，那权重就是“乐手多年训练形成的肌肉记忆”。同一份乐谱，不同熟练度的乐手演奏出来的效果会完全不同。

**伏笔**：知道 LLM 是“续写引擎”还不够，真正关键是它在续写时“凭什么判断下一个词更合理”。下一节就看 Transformer 的内部机制。

## 2.2 Transformer 架构：模型如何「理解」一句话

### Token：把语言切成积木块

AI 不直接处理「文字」，它处理的是「Token」——可以理解为**语言的积木块**。

"我爱北京天安门" 可能被切成：`["我", "爱", "北京", "天安门"]` 四个 Token。

英文 "Hello World" 可能是 `["Hello", " World"]` 两个 Token。

每个 Token 会被转换成一串数字（Embedding，向量），在一个巨大的**语义空间**里找到自己的位置。

Token 也直接影响工程成本：上下文窗口上限、推理时延、以及 API 计费通常都与 Token 数量强相关。

> 🌟 **类比**：想象一个三维空间，"国王"和"女王"的位置很近，"猫"和"狗"的位置很近，"苹果"和"香蕉"的位置很近。这个空间里，意思相近的词，物理位置也相近。这就是 Embedding 的直觉含义。

### Self-Attention：模型的「注意力聚焦」

Transformer 最核心的机制叫 Self-Attention（自注意力）。

考虑这句话：「小明拿起桌上的苹果，他尝试咬了一口，他说____。」

模型在处理「他说____」时，不是只看前一个词，而是会回看整句：

- 「他」会先锁定到「小明」
- 「咬」会强关联「苹果 + 小明」
- 最终在上下文约束下补全「很甜」这类语义更一致的答案

这就是 Self-Attention：**每个词在生成时都会动态计算“与全句其他词的相关性权重”，谁更相关，谁权重更高。**

进一步看，多头注意力（Multi-Head Attention）可以理解为：不是“一个视角”在看整句，而是“多个视角”同时工作。  
有的头更关注主语和代词指代（“他”到底是谁），有的头更关注动作与对象（“咬”对应谁、对应什么），还有的头更关注语气和句法结构（陈述、提问还是补全）。最后把这些视角的结果合并，得到更稳、更全面的理解。

> 🌟 **类比**：你在嘈杂的派对上和朋友聊天。你的大脑会自动过滤掉背景噪音，聚焦在朋友的声音上。Self-Attention 就是这种「选择性聚焦」的能力。

**伏笔**：到这里，模型已经具备“理解并生成”的能力了，但新问题是：模型到底一次能“看见”多少信息？为什么长对话容易失忆？第三章先讲 Context 这张“工作台”。

# 第三章：记忆——上下文工程

先抛一个很多人都会问的问题：**为什么你会感觉模型“记得”你刚才说过的话，甚至能接着你上一个话题继续聊？**

答案并不是模型在这一刻“重新训练”了自己，而是因为系统把你前面的对话内容一并放进了当前请求里。模型看到的是一段被组织好的上下文，于是它能在同一条语境里持续作答，看起来就像“有记忆”。

这一章，我们就把这件事拆开：它到底“记住”了什么、能记多长、为什么会忘、以及怎么通过工程手段让它记得更稳。

下面给一个简化样例，感受一次请求里“上下文”大概长什么样：

```text
[System Prompt]
你是电商平台的技术顾问，回答要简洁，先给结论再给步骤。

[历史对话摘要]
用户正在开发购物车模块，已完成登录和商品列表，当前卡在库存校验。

[最近几轮对话]
User: 我现在要做下单前的库存锁定，应该怎么设计？
Assistant: 建议先做预占库存，再做超时释放...
User: 我用 Redis 做预占，担心并发超卖。

[外部知识 / RAG 片段]
文档A：库存扣减采用“预占 + 最终确认”两阶段策略...
文档B：高并发下推荐 Lua 脚本保证原子性...

[工具返回结果]
压测结果：2000 并发下，超卖率 0.7%，平均延迟 120ms。

[当前用户问题]
那我现在应该优先改哪一块？
```

你会发现，模型并不是“凭空记得”过去，而是在当前这一轮里真的“看到了”这些信息，再据此生成回答。

## 3.1 上下文窗口：模型的「工作台」

想象你的工作桌面。你能同时摊开的文件是有限的，桌面放满了，新的文件就没地方放了。

**Context Window（上下文窗口）** 就是模型的「工作台」。所有当前对话的内容——你说的话、模型的回答、背景资料——都摆在这个工作台上。工作台有大小限制，超出限制的内容，模型就「看不见」了。

单位是 **Token**：

- GPT-3 时代：4K Token（约3000个汉字）
- GPT-4 时代：128K Token（约10万汉字）
- 最新模型：1M Token（约75万汉字，相当于一本厚书）

粗略换算上，1 个英文单词约等于 1.3 Token，1 个汉字也常按约 1.3 Token 估算（会随分词器与文本类型波动）。

> 🌟 **类比**：早期模型像一张小便利贴，你只能写几行字。现在的模型像一张会议室的白板，你能贴上整本产品手册。

## 3.2 「Lost in the Middle」：位置影响注意力

研究发现一个有趣的现象：当你把大量信息塞进上下文时，**模型对开头和结尾的信息记得最清楚，中间的内容容易被「遗忘」**。

这被称为 **Lost in the Middle（迷失在中间）**。

> 🌟 **类比**：这就像开会。主持人说的第一句话和最后总结人们记得最清楚，中间那两个小时的讨论？大家可能只记得几个关键点。

工程实践上，通常有三条简单但有效的策略：

- 把最关键的约束放在开头或结尾；
- 对核心信息做显式编号，必要时适度重复；
- 超长文档优先用 RAG 分块检索，不要整段硬塞进上下文。

有了工作台，下一步就是“往这张工作台上放什么、怎么放”。第四章我们进入 Prompt 工程：把意图翻译成模型能稳定执行的指令。

# 第四章：接口——提示词工程

## 4.1 Prompt 是人与模型的「合约语言」

你有没有遇到过这样的情况：问同一个问题，措辞不同，得到的答案质量差异巨大？

这是因为语言模型**极度依赖输入的质量**。你怎么问，决定了它怎么答。

**Prompt（提示词）** 就是你给模型的「任务说明书」。写得越清楚，模型完成任务的质量就越高。

> 🌟 **类比**：你让一个新实习生帮你写报告。你说「写个报告」，他可能交上来一份你完全不想要的东西。但如果你说「帮我写一份面向非技术高管的季度产品进展报告，500 字以内，分三段：本季亮点、遇到的挑战、下季计划」——他一定会给你更满意的结果。

但“会写 Prompt”不是二元开关，它也有层级。下一节我们按 Zero-shot、Few-shot、CoT 三种形态拆开看。

## 4.2 提示词的三种进化形态

### Zero-shot：直接问

```
用户：把这段话翻译成英文：我今天很开心
```

没有例子，直接让模型完成任务。适合简单任务。

### Few-shot：给几个例子

```
用户：
情感分析：
- 今天天气真好 → 积极
- 这家店服务太差了 → 消极
- 电影一般般 → 中性

请分析：这款产品性价比超高！→
```

通过 2～5 个例子「示范」期望的输出格式，模型会举一反三。

### Chain-of-Thought（思维链）：让模型一步步思考

```
用户：一个班有30名学生，其中40%是女生。
      女生中有一半喜欢数学。请问喜欢数学的女生有几人？
      请一步步思考。
```

加上「请一步步思考」这几个字，在复杂推理任务上通常能带来更稳定的正确率（具体提升幅度取决于模型、题型和评测方式）。

> 🌟 **原理**：这类似于让人在做题时「写出解题过程」。逼迫自己外显化推理步骤，会减少跳步导致的错误。

即便提示词写得很好，模型也可能“语气不稳、边界漂移”。这时就需要看不见但更关键的一层：System Prompt。

## 4.3 系统提示：给模型装「人格芯片」

现代 AI 产品都有一个用户看不见的 **System Prompt（系统提示）**，它定义了模型的身份、规则和行为边界。

```
系统提示（用户看不见）：
  你是一名专业的客服代表，服务于一家高端酒店。
  语气：温暖、专业、耐心。
  规则：不讨论竞争对手，不透露内部折扣政策。
  格式：回答控制在100字以内。

用户：你们有没有更便宜的房型？
模型：[基于系统提示，给出符合酒店品牌调性的回答]
```

系统提示就像给演员的「剧本说明」，告诉他这场戏的背景、角色性格和表演边界。

在真实产品里，用户只看到对话界面，看不到系统提示本体；但模型输出的语气、长度、风格边界，往往都来自这段“隐藏剧本”。

**伏笔**：到这里，模型已经“能回答、会按角色回答”了，但新问题来了——它有时会一本正经地胡说八道，特别是在复杂推理题上。第五章我们用“深度思考”来处理这个问题。

# 第五章：思维——深度思考（Extended Thinking）

## 5.1 两种思维模式

诺贝尔经济学奖得主丹尼尔·卡尼曼在《思考，快与慢》中描述了人类的两套思维系统：

- **系统一（快思考）**：自动的、直觉的、闪电般的。「2+2=？」你不需要思考，答案就冒出来了。
- **系统二（慢思考）**：深思熟虑的、费力的、逐步推理的。「17 × 24=？」你需要拿笔算。

早期的 LLM，更像是**纯系统一**——极快速地生成下一个 Token，但没有真正的「推理过程」。

深度思考模型（如 Claude 的 Extended Thinking、OpenAI 的 o1 / o3）则引入了**类系统二的机制**：在给出最终答案之前，先在内部进行一段「草稿式推理」。

概念层面懂了还不够，下一节我们用一道平均速度题，直观看看“快思考为什么会错，慢思考如何修正”。

## 5.2 思维链的工作方式

想象模型在解一道复杂的数学题：

**没有深度思考（系统一模式）：**

```
问题：一辆车以60km/h行驶2小时，然后以90km/h行驶1.5小时，
      求平均速度。

模型直接输出：75 km/h（错误！直接平均了两个速度）
```

**有深度思考（系统二模式）：**

```
[内部思考过程，用户通常看不见或折叠显示]
  - 总路程 = 60×2 + 90×1.5 = 120 + 135 = 255 km
  - 总时间 = 2 + 1.5 = 3.5 小时
  - 平均速度 = 255 ÷ 3.5 ≈ 72.9 km/h
  
最终输出：约 72.9 km/h ✓
```

> 🌟 **类比**：深度思考就像让模型在答卷之前先打草稿。草稿里可以写错、划掉、重来，最终誊写到答卷上的是经过验证的答案。

但“打草稿”还不是终点，真正稳定来自“交卷前再审一次”。下一节进入自我反思机制。

## 5.3 自我反思机制

更先进的深度思考模型还会进行**自我反思（Self-Reflection）**：

```
[内部思考]
  等一下，我刚才的计算好像有问题。
  让我重新检查一遍……
  对，第二步算错了，应该是这样……
  现在重新验算：✓
```

模型会自我检查、自我纠正，就像一个有自我审核意识的学生在交卷前会主动检查答案。

当然，反思不是免费的。更准，通常也更慢、更贵。下一节我们把这笔账算清楚。

## 5.4 深度思考的代价

当然，世上没有免费的午餐：


| 维度       | 普通回答      | 深度思考                | 差异提示      |
| -------- | --------- | ------------------- | --------- |
| 响应速度     | 秒级        | 十秒到数分钟              | 慢得多       |
| 推理准确率    | 中等        | 显著更高                | 提升明显      |
| Token 消耗 | 低         | 高（思考过程也消耗 Token）    | 常见约 5～10× |
| 适用场景     | 日常对话 / 问答 | 复杂推理 / 代码 / 数学 / 决策 | 高难度任务优先   |


选择深度思考，就像选择「慢炖」而不是「微波炉」——更费时间，但结果更好。实践里更推荐把它当作**任务分级开关**：高复杂度、低容错任务开启；低复杂度、高实时任务关闭。

**伏笔**：即便你把“思考”这件事做好了，模型仍会受到知识时效和私有数据边界限制。下一章我们进入知识问题：RAG。

---

# 第六章：知识——RAG（检索增强生成）

## 6.1 为什么模型的「记忆」不够用

大语言模型的知识，来自训练时的数据。训练结束后，知识就**冻结**了——这叫**知识截止（Knowledge Cutoff）**。

知识截止之后发生的事？模型默认不知道或不可靠。
你公司内部的私有文档？模型更不可能知道。

另外，即使是训练数据范围内的知识，模型也可能**幻觉（Hallucination）**——用听起来很有道理但实际上是编造的内容来回答你。

> 🌟 **类比**：模型就像一个博览群书但与世隔绝的学者，他读过2024年之前所有公开出版的书，但你今天的报纸他没读过，你家的私人日记他也没读过。而且有时候，他会自信满满地给你讲一段他「记忆模糊」编出来的历史——这就是幻觉。

## 6.2 RAG 的工作流程

**RAG（Retrieval-Augmented Generation，检索增强生成）** 的核心思想：**用的时候再去查，查完再回答。**

完整流程如下：

```
用户提问
    ↓
① 将问题转成向量（Embedding）
    ↓
② 在向量数据库中搜索相似内容（语义检索）
    ↓
③ 取出最相关的几段文档
    ↓
④ 将文档 + 原始问题 一起送给 LLM
    ↓
⑤ LLM 基于检索到的真实内容生成答案
    ↓
用户收到有出处的答案
```

> 🌟 **类比**：RAG 就像给学者配了一个「图书管理员助理」。每次有人提问，助理先跑去图书馆查最相关的书，把相关页面复印好，摆在学者面前，学者再根据这些真实资料来回答——而不是凭记忆瞎猜。

到这里，第六章核心就一件事：**让模型在回答前先“查资料”，再“基于资料回答”**。  
它解决的是“知道什么”的问题，让答案更贴近事实、更新、更可追溯。

**伏笔**：走到这里，模型本质上还只是“会说”——能基于资料给出更像样的答案，但还不会真正去执行现实动作。下一章我们进入 Tool：让 AI 从“会答”迈向“会做”。

# 第七章：能力——Tool（工具调用）

## 7.1 模型走出「沙箱」的关键一步

在没有 Tool 的时代，语言模型是一个孤岛。

你可以问它「现在几点了？」，它只能说「我无法获取实时时间」。
你可以问它「帮我发一封邮件」，它只能给你生成邮件内容，但无法真的发出去。

**Tool（工具调用）** 打破了这个孤岛。模型终于可以**调用外部世界的能力**：查天气、执行代码、读写数据库、发邮件……

> 🌟 **类比**：没有 Tool 的 AI，像一个只能「说」不能「做」的人。有了 Tool，他就像一个有手机的人：想查资料可以搜索，想转账可以操作手机银行，想打车可以叫滴滴——他的「执行半径」大幅扩展。

## 7.2 Function Calling：模型学会「点菜」

技术上，Tool 是通过 **Function Calling（函数调用）** 实现的。

开发者提前告诉模型：「我有这些工具可以用——」

```json
工具列表：
1. get_weather(city) → 查询城市天气
2. send_email(to, subject, body) → 发送邮件
3. search_db(query) → 查询数据库
```

当用户说「帮我查一下明天北京的天气，然后发邮件给老王」，模型不会直接生成回答，而是先「点菜」：

```json
模型输出（不是给用户看的，是给系统看的）：
{
  "tool": "get_weather",
  "params": { "city": "北京", "date": "明天" }
}
```

程序接收到这个指令，真的去查天气，把结果返回给模型，模型再决定下一步：

```json
{
  "tool": "send_email",
  "params": {
    "to": "laowang@company.com",
    "subject": "明天北京天气提醒",
    "body": "老王，明天北京预计晴天，温度18-26°C，适合外出。"
  }
}
```

> 🌟 **类比**：模型像一个项目经理，它不亲自去执行每个任务，而是**识别需求 → 找对应的工具/专家 → 发出结构化指令 → 整合结果 → 输出最终答案**。

## 7.3 工具的设计原则

**单一职责**：一个工具只做一件事。`send_email` 只发邮件，不兼职查天气。模块化、可复用。

**幂等性**：同一个工具调用多次，结果应该一样（至少不会造成灾难）。查天气调用两次没问题，但如果「发邮件」的工具不是幂等的，模型重试一下就发了两封邮件……

**错误处理**：工具调用失败了怎么办？模型应该知道失败原因，然后决定是重试、换个工具、还是告诉用户。

# 第八章：协议——MCP（Model Context Protocol）

## 8.1 「工具碎片化」的困境

当 Tool 能力越来越丰富，一个新问题就会出现：**同一能力，需要为不同模型重复接入**。

假设你要把 GitHub、Notion、Slack 三个能力接进 AI 应用：

- 对接 A 模型，要写一套函数定义格式；
- 对接 B 模型，要改成另一套参数格式；
- 对接 C 平台，还要重写鉴权和错误映射。

结果就是：业务逻辑没变，适配代码越堆越多。每次 API 升级、权限策略变化，三四套代码都要跟着改。

> 🌟 **类比**：这就像一家公司同时用多种插座标准。咖啡机本身没变，大家每天却在找转换头。

## 8.2 MCP 解决了什么：统一契约 + 模型无关

**MCP（Model Context Protocol）** 本质上不是“某家模型的新功能”，而是 AI 与外部能力之间的一层**标准通信契约**。

它解决的是“接入层标准化”问题：工具开发者把能力实现成 MCP Server，支持 MCP 的 AI Client 都能按同一方式发现和调用。

```
┌─────────────────────────────────────────────────┐
│                  AI 应用（Client）               │
│          (Claude / GPT / Cursor / ...)          │
└──────────────────────┬──────────────────────────┘
                       │ MCP 协议（统一语言）
        ┌──────────────┼──────────────┐
        ↓              ↓              ↓
   GitHub MCP      Notion MCP    Slack MCP
   Server          Server        Server
```

一句话记忆：**你不再“给每个模型单独接工具”，而是“把能力接进协议生态”。**

## 8.3 MCP 的工作方式：不只 Tools

很多人把 MCP 理解成“新的函数调用”，但它覆盖的是三类能力：


| 类型            | 回答的问题       | 典型例子              |
| ------------- | ----------- | ----------------- |
| **Tools**     | AI 可以做什么    | 创建 Issue、发消息、执行操作 |
| **Resources** | AI 可以看什么    | 文件内容、数据库记录、实时状态   |
| **Prompts**   | AI 可以复用什么模板 | 评审模板、报告模板、标准任务入口  |


这三者配合起来，才构成一个完整的“外部能力平面”：既能做事，也能读数据，还能复用最佳实践入口。

### 8.3.1 MCP 的渐进式披露与按需加载

MCP 在工程上也很适合做“**渐进式披露 + 按需加载**”，而不是把所有 Server 的全部能力说明一次性塞进上下文。

一个更稳的加载链路通常是：

1. **先注入能力索引**：只给模型看精简目录（有哪些 Server、有哪些 Tool/Resource/Prompt、用途标签）；
2. **再按任务筛选候选**：根据当前意图，仅保留相关能力；
3. **命中后再展开细节**：只在需要时加载目标工具的 schema、参数约束、示例与资源片段；
4. **执行后回注最小结果**：把必要结果回写上下文，而不是整包原始返回。

这样做的收益和 Skill 的加载思路一致：

- **控成本**：减少无关能力描述占用 token；
- **提稳定**：上下文更干净，模型路由与参数选择更少跑偏；
- **便治理**：按任务域逐步暴露能力，更容易做权限边界和审计。

## 8.4 两种技术流派：CLI 流派 vs MCP 流派

到工程落地时，常见两条路线：

- **CLI 流派**：直接让 Agent 调 shell/命令脚本，强调“快接入、低门槛、先跑通”；
- **MCP 流派**：把能力协议化暴露给模型，强调“可发现、可复用、可治理”。

这不是“谁淘汰谁”的关系，而是两个层级在解决不同问题。


| 维度     | CLI 流派（终端命令/脚本） | MCP 流派（协议化能力）          |
| ------ | --------------- | ---------------------- |
| 启动成本   | 低，单人可以快速起步      | 中等，需要设计 Server 契约      |
| 接口一致性  | 依赖命令习惯，格式易分裂    | 结构化参数与能力协商，接口更稳定       |
| 跨客户端复用 | 弱，迁移常靠脚本改造      | 强，一次接入可被多 Client 使用    |
| 治理与审计  | 常分散在脚本和运维规范中    | 更容易在 Server 层做权限、日志、边界 |
| 最佳场景   | 原型期、一次性任务、个人效率  | 生产期、多团队协作、长生命周期系统      |


一个常见误解是“用了 MCP 就不能用 CLI”。其实不是：

- 本地 MCP Server 往往就是通过 stdio 跑起来的；
- Server 内部也完全可以封装 CLI 命令；
- MCP 约束的是“**对模型暴露什么契约**”，不是“禁止命令行”。

**实用选型原则：**

1. 单人、短周期、低风险任务，优先 CLI 快速验证价值；
2. 一旦进入多团队复用、高风险动作、跨应用协作，优先 MCP；
3. 最稳妥的迁移路径是：先 CLI 验证，再把高频能力协议化成 MCP。

## 8.5 工程价值与落地建议

MCP 真正的价值，不在“会不会调用工具”，而在三件工程结果：

- **复用**：同一能力不再反复适配；
- **治理**：权限、参数、审计边界集中在 Server 层；
- **迁移**：模型或客户端切换时，业务接入层改动更小。

落地建议也很朴素：不要一上来全量改造。优先挑 2～3 个高频且稳定的能力先 MCP 化（如代码托管、文档系统、消息系统），跑通后再扩面。

## 8.6 生态现状

MCP 提出后，IDE 与 SaaS 工具都在加速跟进，行业也逐步形成“协议优先”的共识。

> 🌟 **比喻**：MCP 不只是“多一个接口”，它更像 AI 时代的 USB 规范——单点价值有限，生态价值巨大。

# 第九章：智能体（AI Agent）

## 9.1 AI Agent 在做什么

有了 **Tool** 和 **MCP** 之后，模型已经能**调用外部能力**；**Agent** 要解决的下一层问题，是**怎么在多轮里连续决策**，而不是「问一句、答一句就结束」。
这里回扣第一章：**Agentic AI 是“目标导向的自主完成”这一范式，AI Agent 是落地该范式的一种工程实现。**

一个常见的 **AI Agent** 实例可以想成：用 LLM 当「大脑」，在每一轮里决定**下一步要调哪个工具、传什么参数**，看清返回结果，再决定要不要继续，直到停在你设定的结束条件。  
它仍然是普通软件/产品形态，不额外叠概念。

日常实现里，大家会在下面几条能力上做组合（不必一次全有）：

- 多轮里持续对齐用户目标，而不是单轮就散场；
- 通过 Tool / MCP 读网页、调 API、改本地文件，把「说」和「做」接起来；
- 任务拆小步、某步失败就改策略或重试（而不是硬编一句输出）。

## 9.2 感知 → 规划 → 执行

```
         外部世界
    (网页 / 文件 / 邮件 / 数据库...)
            ↓ 感知（Perceive）
         ┌──────┐
         │ LLM  │ ← 当前状态 + 历史记忆
         └──────┘
            ↓ 规划（Plan）
      分解任务、决定下一步行动
            ↓ 执行（Act）
      调用工具 / 写入文件 / 发送消息...
            ↓
       观察结果 → 反馈回 LLM
            ↓
    [循环，直到任务完成]
```

这就是 **Agent Loop**：**感知 → 规划 → 执行 → 看结果 → 再决策**。**Tool** 是里面「动真格」的那一环。

## 9.3 ReAct：想一步、做一步

**ReAct（Reasoning + Acting）** 只是把上面的循环，用「思考 / 行动 / 观察」写清楚，便于对照日志排查。

```
任务：帮我查找 "Model A" 的最新价格，
      并和 "GPT-4o" 做对比表格。

模型思考：我需要查两个模型的价格，分别搜索。
行动：[调用搜索工具] search("Model A official pricing")
观察：返回结果：输入价格 X，输出价格 Y（示意）
模型思考：Model A 价格有了，现在查 GPT-4o。
行动：[调用搜索工具] search("GPT-4o official pricing")
观察：返回结果：输入价格 P，输出价格 Q（示意）
模型思考：两个价格都有了，现在生成对比表格。
最终输出：[一张格式清晰的价格对比表]
```

> 注：价格信息变化快，正式文档或汇报请以各模型官方定价页面的最新数据为准。

Agent 会循环、会调工具之后，新的关键问题是：同一类事怎么做才标准、怎么验收。下一章进入 **Skill（技能说明）**。

---

# 第十章：技能——Agent Skills

## 10.1 从“会调用”到“会完成”：为什么还需要 Skill

第七章到第九章，已经让 AI 具备了“能查、能调、能循环执行”的能力。  
但业务里真正棘手的问题往往是：**同一种任务，换一个人/换一个提示词，结果质量就飘**。

这就是 Skill 存在的意义：它不是替代 Tool，也不是替代 Agent，而是把“怎么做”沉淀成一套可复用的执行手册。

RAG 解决「知道什么」，Tool/MCP 解决「能做什么」，Agent 解决「多轮怎么推进」；**Skill 解决「怎么按同一标准完成」**。

同时，Skill 还解决了一个常被忽略的上下文问题：不是把所有操作规范一次性塞进系统提示词，而是通过**渐进式披露（先注入技能索引）+ 按需加载（命中后再展开步骤）**，把真正相关的方法在需要时再放进当前上下文。这样既能降低 token 压力，也能减少长上下文里的噪音干扰，让模型把注意力集中在当前任务上。

> 🌟 **类比**：RAG 是资料室，Tool 是工具间，Agent 是执行者，Skill 是岗位 SOP。没有 SOP，执行者再能干也容易风格漂移。

## 10.2 一个 Skill 的关键构成：触发 + 步骤 + 输出

一个真正好用的 Skill，通常要把三件事写清楚：

- **触发条件（description）**：什么场景应该调用这个 Skill；
- **执行步骤（workflow）**：先做什么、后做什么、遇到异常怎么办；
- **输出格式（template）**：结果应该长什么样，如何验收。

以“代码 Review Skill”为例，真正影响效果的不是“术语多不多”，而是这些细节是否明确：

- 触发语境：看到“评审/审查/质量检查/安全检查”等请求即触发；
- 步骤顺序：先理解意图，再按严重级别排问题，最后给可执行建议；
- 输出约束：固定分组与字段，避免每次结构都不一样。

这也是为什么 Skill 的正文要尽量写成“可执行指令”，而不是概念口号。模型不是来读理念的，它是来执行任务的。

## 10.3 Skill 生命周期：起草 → 测试 → 迭代

很多团队写 Skill 失败，不是因为不会写，而是把它当成“一次性文档”。  
更有效的做法，是把 Skill 当成一个持续迭代的产物：

1. **起草**：先写最小可用版本，明确触发、步骤、输出；
2. **测试**：用 2～3 个真实用户请求做小样本验证；
3. **迭代**：根据失败样例改 description 和步骤，而不是只改措辞；
4. **扩展**：稳定后再扩场景，沉淀成团队标准。

这个过程和软件开发很像：不是“写完就结束”，而是“上线后持续优化命中率和输出质量”。

## 10.4 Skill 资产库：团队经验的结构化沉淀

当团队有了几十个可复用 Skill，收益会从“个人效率”升级为“组织能力”：

- 新人上手速度更快：减少“口口相传”的隐性知识损耗；
- 交付质量更稳定：不同人执行同类任务，输出更一致；
- 复盘成本更低：问题可以定位到“触发不准”还是“步骤不全”。

常见 Skill 资产包括：

- PRD 写作 Skill；
- 技术方案评审 Skill；
- 周报与复盘 Skill；
- 故障通报 Skill。

## 10.5 何时用 Skill，何时用 MCP

可以用一句话区分：

- **Skill** 解决“这件事应该怎么做才标准”；
- **MCP** 解决“这件事能不能稳定连到外部系统去做”。


| 场景特征                             | 优先选择               |
| -------------------------------- | ------------------ |
| 任务是流程标准化问题（写法、步骤、结构统一）           | **Skill**          |
| 任务需要连接外部系统（GitHub/Slack/DB）并执行动作 | **MCP**            |
| 既要统一做法，又要连外部能力                   | **Skill + MCP 组合** |


在真实工作里，可以这样快速判断：

- **自然语言生成大数据 SQL（数仓分析）**：优先 **Skill + MCP 组合**。  
Skill 负责沉淀统一口径（时间范围、指标定义、维度命名、SQL 模板与自检清单），MCP 负责连接 Hive/Spark/Trino 等查询引擎并执行 SQL。
- **研发流程里创建 Redmine 任务**：优先 **MCP**（若团队有固定规范则升级为组合）。  
纯动作层面是“调用 Redmine API 创建 issue”；如果你还要求统一字段映射（优先级、模块、验收标准模板），就用 Skill 先约束再调用 MCP。
- **创建分支（Git Flow）**：优先 **Skill + MCP 组合**。  
Skill 约束命名规则与流程（从哪个基线切、是否关联任务号、何时推送远端），MCP/工具层执行真实 git 动作。
- **Code Review**：通常分两层。  
评审标准与输出结构（严重级别、风险分类、结论模板）属于 Skill；拉取 PR、读取 diff、回写评论属于 MCP，所以在团队协作场景里多数是组合。

给一张“10 秒选型小抄”：

1. **只需要统一做法，不需要连外部系统** → 先用 **Skill**；
2. **只需要稳定执行外部动作，不强调过程模板** → 先用 **MCP**；
3. **既要过程可复用，又要真实执行动作** → 用 **Skill + MCP 组合**（默认推荐）。

实践里最常见的组合方式是：**Skill 定义流程，MCP 提供动作**。

例如“发版复盘 Skill”可以规定：

1. 先读发布记录（调用 MCP Resource）；
2. 再抓告警与指标（调用 MCP Tool）；
3. 最后按固定模板输出结论（Skill 输出格式）。

这样，你既有统一方法论，又有真实可执行能力。

当 Skill、MCP、Agent 开始规模化协作，你就会遇到调度、恢复、权限、观测问题。下一章进入 Harness。

# 第十一章：工程化——Harness（运行时层，Agent Runtime Layer）

## 11.1 先把名字讲清楚：Harness 到底是什么

很多团队把 Harness 理解成“某个 Agent 框架的名字”，其实本文里的 Harness 指的是一个**运行时层抽象（Runtime Layer）**，不特指某个单一产品：

- 上层接用户目标与业务约束；
- 下层接模型、工具、MCP、存储、权限系统；
- 中间负责把一次“会话式回答”变成“可持续执行的任务流程”。

一句话定义：**Harness = 让 Agent 在真实生产环境里稳定运行的工程底座。**

如果没有 Harness，Agent 常见症状是：

- 跑一半崩了只能重来；
- 多 Agent 协作互相打架；
- 出了事故找不到谁在何时调用了哪个高风险工具；
- 迭代一段时间后，系统可用但不可管、可跑但不可审。

> 🌟 **类比**：Agent 是“会干活的人”，Harness 是“让团队可管理地干活的公司制度与基础设施”。

## 11.2 Harness 的核心能力模型（重点）

真正有工程价值的 Harness，至少要同时覆盖 6 个能力面：

1. **编排（Orchestration）**
  负责把目标拆成步骤，控制单 Agent 或多 Agent 的执行顺序、并发策略和终止条件。
2. **状态（State）**
  维护任务上下文、阶段结果、失败原因与中间产物，让任务可恢复而非“一次性对话”。
3. **工具与协议（Tools + MCP）**
  统一管理可用能力、参数约束、鉴权边界，避免“谁都能直接调任何高危动作”。
4. **安全与治理（Guardrails）**
  做权限分级、策略拦截、人工审批、预算控制，防止自动化越界。
5. **可观测性（Observability）**
  记录轨迹、调用链、重试与失败点，支持实时监控、离线复盘和审计。
6. **评估与迭代（Evaluation Loop）**
  通过离线回放、线上指标、失败样本分析持续改进 Prompt、Skill、Tool 路由策略。

你可以把它看成一个闭环：

```
用户目标
   ↓
编排执行（单Agent/多Agent）
   ↓
调用 Tools / MCP / 记忆
   ↓
策略校验与审批
   ↓
结果产出与日志沉淀
   ↓
复盘评估 → 反哺下一轮策略
```

### 多 Agent 协作为什么离不开 Harness

当任务进入“后端 + 前端 + 测试 + 文档”并行阶段，真正困难的不是生成内容，而是下面这些运行时问题：

- 子任务依赖如何表达；
- 某个分支失败后是重试、降级还是回滚；
- 多路结果如何合并并校验一致性；
- 谁有权限触发生产级动作。

这些都属于 Harness 的责任，而不是某个单独 Agent 的“聪明程度”。

## 11.3 主流 Harness 方案观察（按场景适配）

下面这张表按“工程落地视角”给出观察维度，不按“谁更强”做绝对排序；具体结论需结合团队约束与试点评估：


| 方案/平台                            | 核心定位         | 观察维度（示例）         | 适配条件                |
| -------------------------------- | ------------ | ---------------- | ------------------- |
| **IDE 内嵌型 Agent 平台**（如 Cursor 类） | 以开发者交互效率为核心  | 代码理解链路、协作体验、工具生态 | 适合日常研发协作与快速迭代       |
| **流水线型 Coding Agent 体系**         | 以批量任务自动执行为核心 | 执行吞吐、可回放性、失败恢复   | 适合重复性工程任务与自动化流水线    |
| **开放编排型 Agent 框架**               | 以可扩展与私有化为核心  | 可定制性、插件体系、运维复杂度  | 适合多系统集成与企业定制场景      |
| **多角色协作型 Agent 架构**（如 Hermes 类）  | 以任务分治与协同为核心  | 路由策略、状态管理、治理能力   | 适合复杂流程拆解与多 Agent 协作 |


### 选型建议（实战版）

- **先看约束，不先看名气**：数据合规、私有化要求、审计强度通常比“模型效果”更先决定方案。
- **先选运行模式，再选产品**：你要的是 IDE 辅助、流水线自动化，还是多 Agent 编排平台，三者路线不同。
- **先小范围验证闭环**：优先挑一个高频、可量化场景做 PoC（例如“自动代码评审 + 问题单回写”），验证可观测与治理链路。
- **保留协议层可迁移性**：尽量把外部能力抽象在 MCP/工具层，避免把核心业务强绑定在单一平台私有接口上。

## 11.4 Human-in-the-Loop：自动化边界怎么划

Harness 的成熟度，最终体现在“自动化边界”是否清楚。一个实用分层是：

```
低风险、可逆操作 → 自动执行
  例：检索资料、生成草稿、创建测试分支

中风险、可回滚操作 → 建议人工确认
  例：合并代码、发送外部通知、修改共享配置

高风险、不可逆操作 → 强制人工审批
  例：生产部署、批量删除、资金相关动作
```

当你把这条边界和日志审计、权限策略、失败回放打通，Harness 才算真正进入“可生产”的阶段。

# 第十二章：Hermes 技术原理深度解析

## 12.1 为什么 Hermes 值得单独拆解

前一章我们讲的是 Harness 的通用方法论，这一章看一个有代表性的实现：**Hermes Agent**。

它有意思的地方，不是“又一个 Agent 产品”，而是把三件事情放在了架构核心：

- 让 Agent 不只是执行，还能持续学习；
- 让记忆不只是“可选调用”，而是有结构保证；
- 让智能层与执行层解耦，兼顾效率与安全。

一句话理解 Hermes：**它把 Agent 从“会做事”推进到“会积累做事方法”。**

## 12.2 三层架构：接入层、智能层、执行层

Hermes 的整体结构可概括为三层：

1. **接入层（Gateway）**
  负责多渠道消息接入、会话归一化、身份与权限控制。
2. **智能层（AIAgent Core）**
  负责 Prompt 组装、模型路由、Skill 检索、记忆读取、任务调度。
3. **执行层（Backends）**
  负责终端、浏览器、文件、MCP 等真实动作执行，可本地或远端隔离运行。

这个分层的工程价值是：**能力可以协同迭代，风险可以分层隔离**。智能层升级不必强耦合执行环境；执行后端扩展也不必重写核心 Agent 循环。

## 12.3 Hermes-Agent（个人智能体） 的核心工程设计

### 设计一：闭环学习（Closed Learning Loop）

多数 Agent 在任务结束后“经验蒸发”，Hermes 重点做的是“任务完成后还能沉淀可复用能力”。

它的目标不是只把这次任务答对，而是让下一次相似任务更快、更稳。

### 设计二：四层记忆体系（而非单一上下文注入）

Hermes 把记忆拆成不同职责层：

- 长期稳定事实（如环境约束、项目偏好）；
- 用户画像偏好；
- 可检索的会话档案（FTS 检索）；
- 可扩展的外部记忆插件。

这种设计避免了“把所有记忆都塞进系统提示”造成的膨胀问题，也降低了“只靠临时上下文导致遗忘”的风险。

### 设计三：Skill 的系统提示词注入与分层加载

你提到的关键点非常重要：**Skill 最终确实要作用到系统提示词（system prompt）里**，但 Hermes 不会粗暴地把所有 Skill 文档全量塞进去，而是做了“分层注入 + 按需展开”。

一个典型加载链路可以理解为：

```
消息到达
  ↓
Prompt 构建器先组装基础层
  （角色设定 + MEMORY + USER + 安全规则）
  ↓
注入 Skill 候选索引（名称/描述/路径/标签）
  ↓
模型根据当前任务选择命中的 Skill
  ↓
按需读取该 Skill 的摘要或正文片段
  ↓
将命中 Skill 片段拼入当前系统上下文
```

这样做的工程原因是三条：

1. **控制 token 成本**：避免 Skill 库越大，系统提示词越失控；
2. **保持缓存稳定**：基础提示前缀更稳定，命中缓存的概率更高；
3. **降低误触发风险**：先让模型“选 Skill”，再“读全文”，比一股脑注入更可控。

进一步看，Hermes 在 Skill 管理上通常会拆成三层：

- **索引层**：技能元数据（触发描述、标签、优先级）；
- **内容层**：Skill 正文（步骤、边界、输出格式）；
- **执行层**：可选脚本/工具调用约束（必要时与 MCP/Tool 协同）。

这和第十章讲的原则一致：`description` 负责“何时触发”，正文负责“如何执行”，不是把 Skill 当静态文档摆在那里。

在 Hermes 一类实现里，这套流程可抽象为“先找、再看、后管”的最小闭环：先枚举候选 Skill，再按需读取内容，最后在需要时回写和治理。

一个简化时序如下：

```
用户任务到达
  ↓
Agent 调用 skill_list（获取候选技能）
  ↓
模型判断命中后调用 skill_view（读取目标 Skill 内容）
  ↓
命中片段注入当前系统上下文并执行
  ↓
任务结束后调用 skill_manage（必要时更新/修补 Skill）
```

### 设计四：Skill 自动生成与迭代流水线

这是 Hermes 最差异化的一点：它不只消费 Skill，还会在复杂任务后尝试自动提炼 Skill，并在后续任务中修补和复用。

可抽象成五步：

```
执行任务 → 评估是否值得沉淀 → 提取结构化步骤
→ 后续任务中持续修补 → 命中相似任务时直接复用
```

这让 Skill 从“手工知识库”变成“半自动进化资产库”。代价是 token 与治理成本会上升，但长期收益是团队经验累积速度明显提高。

### 设计五：MCP 接入的技术基础与优化策略

Hermes 把 MCP 当成“能力总线”接入，而不是“能连就行”的插件。核心流程通常是：

```
启动阶段：发现并注册 MCP Server 能力
  ↓
运行阶段：按任务意图路由到目标 MCP Tool/Resource
  ↓
执行阶段：参数校验、鉴权、调用、结果标准化回注
  ↓
失败阶段：重试/降级/切换后端并记录可观测日志
```

相比“直接把所有工具描述扔给模型”，Hermes 类实现常见的优化点有：

- **能力索引缓存**：缓存 Server 的工具清单与 schema，减少每轮重复拉取；
- **按需暴露**：根据任务域只暴露相关工具，降低上下文噪音；
- **参数预校验**：在执行前做 schema 校验，减少无效调用；
- **调用去重与幂等保护**：避免模型重试导致重复副作用；
- **结果标准化**：不同 MCP Server 返回结构统一包装，方便模型稳定消费；
- **超时与熔断**：慢工具自动降级，防止单点拖垮整轮执行。

这些优化的本质是：**把“模型的不确定性”隔离在调度层，把“系统的确定性”落实在协议和运行时层。**

### 设计六：智能层与执行层物理分离

Hermes 支持把“思考”与“执行”放在不同环境：例如 Agent 在一台主机推理，工具执行在隔离容器或远端计算节点。

对工程团队来说，这个设计的价值是：

- 安全边界更清晰；
- 高性能资源利用更灵活（如远端 GPU/集群）；
- 合规场景更容易做权限收口和审计。

## 12.4 与通用 Harness 方法论的映射

把 Hermes 放回第十一章的 Harness 框架，你会看到一一对应关系：


| Harness 能力面 | Hermes 的体现         |
| ----------- | ------------------ |
| 编排          | 任务循环 + 多步骤工具执行     |
| 状态          | 会话持久化与结构化上下文管理     |
| 工具协议        | 原生工具体系 + MCP 可插拔接入 |
| 安全治理        | 分层权限与执行隔离          |
| 可观测性        | 会话轨迹、工具链路、状态留痕     |
| 评估迭代        | 自动 Skill 提炼与持续改进   |


所以 Hermes 的价值，不在于“概念更新”，而在于它把“Agent 工程化闭环”做得更前置。

## 12.5 适用场景与边界

Hermes 尤其适合：

- 重复性复杂任务（需要长期积累方法）；
- 多工具联动的工程流程；
- 对可持续优化有要求的研发团队。

但也要看到边界：

- 体系更强意味着配置与治理复杂度更高；
- 自动沉淀机制需要质量门禁，避免“坏经验自动放大”；
- 在极简、一次性任务里，重框架可能得不偿失。

**过渡建议**：先把 Hermes 用在 1～2 条高频流程上，验证“沉淀收益 > 维护成本”，再扩到更多业务线。

# 第十三章：全景回顾与未来展望

## 13.1 从“能力堆叠”到“结果闭环”

如果把全文压缩成一句话，就是：

**我们不是在堆技术名词，而是在搭一条从“理解需求”到“稳定交付结果”的闭环流水线。**

前半段（第 1～6 章）解决“会理解、会思考、会补知识”；  
中段（第 7～10 章）解决“会调用能力、会按方法执行”；  
后段（第 11～12 章）解决“能稳定运行、能持续进化”。

于是，AI 系统能力不再是“单点回答质量”，而是“端到端结果质量”。

## 13.2 一张图串联所有层

```
【用户意图】
     ↓
【Prompt 工程】← 把意图表达清楚
     ↓
【上下文工程】← 工作台整理好
     ↓
【深度思考】← 复杂问题先打草稿
     ↓
┌────────────────────────────┐
│         核心 LLM            │
│     （思考 + 规划 + 生成）   │
└────────────────────────────┘
     ↓            ↓
【RAG 知识库】← 外部知识补全
     ↓
【Tool 调用】← 动手执行
     ↓
【MCP 协议】← 标准化接入工具生态
     ↓
【Agent Loop】← 多步骤自主循环
     ↓
【Skill 技能库】← 可复用执行手册
     ↓
【Harness 框架】← 可靠、可观测、可管理地运行
     ↓
【Hermes 类实践】← 闭环学习与持续进化
     ↓
【最终结果交付】
```

这张图有三个关键关系：

- **纵向依赖**：上层能力依赖下层稳定性，不能跳层建设；
- **横向协同**：RAG、Skill、MCP、Harness 不是替代关系，而是分工关系；
- **闭环迭代**：交付结果会反哺记忆、Skill 与运行策略，系统越跑越稳。

## 13.3 技术演进的三个阶段（重述）

**阶段一：会生成（Generative）**  
核心能力是“语言生成”，价值集中在内容产出。

**阶段二：会执行（Agentic）**  
核心能力是“多步执行”，价值从内容转向任务完成。

**阶段三：会进化（Self-Improving Agents）**  
核心能力是“经验沉淀与策略优化”，价值从一次完成升级为持续变强。

这三个阶段并非彼此替代，而是层层叠加：先会生成，再会执行，最终会进化。

## 13.4 接下来 2～3 年的三个确定趋势

**趋势一：人机分工从“写答案”转向“管系统”**  
人类更多负责目标、边界、验收与审计；AI 负责拆解、执行与回报。

**趋势二：组织竞争力从“模型选择”转向“工程化能力”**  
同样的模型，是否有高质量 Skill、稳定 MCP 能力层、可观测 Harness，结果差异会非常大。

**趋势三：个人与团队都将拥有“Agent 基础设施”**  
个人侧是记忆 + 工具 + 偏好；团队侧是规范 + 权限 + 审计 + 闭环学习。

## 13.5 给实践者的三条落地建议

1. **从高频流程开始，不从全栈改造开始**：先挑一条业务闭环跑通。
2. **优先建设可观测与治理，再追求全自动**：没有审计与边界，自动化会反噬。
3. **把经验产品化**：把有效做法沉淀成 Skill，把稳定能力沉淀成 MCP，把执行策略沉淀进 Harness。

如果你准备在团队里做第一次 PoC，可以用这个最小清单：

- **输入**：选 1 条高频、可重复、可量化流程（例如“代码评审与问题回写”）；
- **输出**：定义可验收交付物（如评审结论模板、问题单字段完整率）；
- **指标**：至少跟踪成功率、平均耗时、人工介入率、回滚率四项；
- **对照**：保留人工基线流程，做 A/B 或前后对比，避免“只看主观体验”；
- **边界**：明确自动执行与人工审批分界，并保留完整日志用于复盘。

到这个阶段，AI 系统才不是“会聊天”，而是真正“会交付”。

## 写在最后

如果你一路读到这里，应该已经能看清这次范式变化的本质：

它不是“模型更聪明了一点”，而是**软件生产方式在重写**。

过去我们优化的是“人如何更高效地使用工具”；  
现在我们要设计的是“工具如何在边界内自主完成目标”。

从 Prompt 到 RAG，从 Tool 到 MCP，从 Agent 到 Skill，再到 Harness 与 Hermes 类闭环实践，这条路线最终指向同一个答案：

**AI 的终局价值，不在回答本身，而在可验证、可复用、可持续优化的结果交付。**

所以，与其问“哪个模型最强”，不如开始搭建你的下一代能力栈：

- 有标准（MCP）；
- 有方法（Skill）；
- 有运行时（Harness）；
- 有进化能力（闭环学习）。

当这些部件开始协同，你就不只是“在用 AI”，你是在构建自己的 AI 生产系统。

这，才是从 Generative AI 走向 Agentic AI 的真正意义。

最后，再给两组更现实的建议：

**给个人：这是软件从业者最好的时代之一。**  
过去一个想法从原型到可用版本，往往要投入大量时间；今天借助模型、Agent 和工具链，很多想法几小时就能看到可运行结果。  
与其过度焦虑“会不会被替代”，不如把注意力放在更实在的问题上：你的实验密度够不够高？你的 token 用得够不够值？你是否在持续把新能力转成自己的工作流资产？

**给团队：AI 时代不能用旧组织方式硬套新生产力。**  
如果还沿用过去的组织结构、管理方式和流程，通常会在执行效率、协同速度和学习迭代上全面掉队。  
真正有效的做法是：主动拥抱先进工具与模型，持续试错、快速复盘、及时固化为团队规范。  
说到底，纸上得来终觉浅，觉知此事要躬行。
