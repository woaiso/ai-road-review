# Skills 与 MCP：从概念到实战的完整解析

---

## 一、什么是 Skills，什么是 MCP

### Skills：给 AI 装上"操作手册"

**Skill 解决的核心问题是"怎么做"。**

想象你刚入职一家公司，公司给了你一位聪明的实习生助手——他学历顶尖、理解能力极强，但他不知道你们团队的报告格式、标准流程、输出规范。你让他"写份竞品分析"，他交上来一份"有自己发挥"的东西，跟预期差距很大，还得返工。

这就是没有 Skill 的 AI 现状。**Skill 本质上是"程序性记忆"**——不是"苹果公司成立于1976年"这类陈述性知识，而是"当需要写竞品分析时，先做市场规模研究，再做功能对比，最后做 SWOT，输出格式为……"这类操作性知识。

一个标准 Skill 是一个目录，核心是 `SKILL.md` 文件：

```markdown
---
name: weekly-report
description: 当用户需要生成、撰写、整理周报时使用此技能。
             可接受原始工作记录（commit记录、Jira任务列表）作为输入，
             输出结构化的周报。
---

## 你的角色
你是用户的工作助手，帮助将一周工作记录整理成结构清晰的周报。
读者是直属上级，关注：完成了什么、遇到什么问题、下周计划。

## 执行步骤
1. 仔细阅读原始工作记录
2. 按「已完成」「遇到的问题」「下周计划」三个板块分类整理
3. 按重要性排序，总长度控制在 300-500 字

## 输出格式
**本周工作汇报（YYYY.MM.DD - YYYY.MM.DD）**
**已完成** / **遇到的问题** / **下周计划**
```

**Skill 最关键的工程机制是"渐进式披露"**：Agent 启动时只加载每个 Skill 的 `name + description`（几十个 Token），只有用户的请求语义命中时，才把完整 SKILL.md 注入上下文。这解决了"技能越多越好，但上下文窗口有限"的根本矛盾。

---

### MCP：给 AI 世界统一"插座标准"

**MCP 解决的核心问题是"能做什么"，以及"怎么以标准化方式接入外部工具"。**

在 MCP 出现之前，同一个 GitHub 工具，要为 Claude、GPT、Gemini 分别写一遍适配代码——参数格式不同、认证方式不同、错误处理不同。每次 API 更新，多套代码都要维护。就像各国插座标准不统一，出差每次都要带一堆转换头。

**MCP（Model Context Protocol）就是 AI 工具世界的"USB 标准"。** 工具开发者只写一次 MCP Server，任何支持 MCP 协议的 AI 应用直接复用。

MCP 覆盖三个维度：


| 维度   | MCP 术语        | 说明                     |
| ---- | ------------- | ---------------------- |
| 执行操作 | **Tools**     | AI 主动触发：创建文件、发消息、调 API |
| 读取数据 | **Resources** | AI 被动访问：读文件、查数据库、获取状态  |
| 使用模板 | **Prompts**   | 预定义的提示词组合，快速完成标准任务     |


底层通信是 **JSON-RPC 2.0**，支持三种传输方式（Stdio 本地进程通信、HTTP+SSE 远程服务、WebSocket 实时双向）。

## 二、Skills vs MCP：优劣对比与实际工作场景

### 一张表先看清楚定位


| 对比维度     | Skills              | MCP                    |
| -------- | ------------------- | ---------------------- |
| **解决问题** | AI 怎么做事（执行方法论）      | AI 能做什么（外部能力边界）        |
| **本质**   | 操作手册 / SOP          | 工具接口 / 插座标准            |
| **加载时机** | 语义命中时动态注入           | 会话启动时建立连接              |
| **是否执行** | 不直接执行，指导 LLM 行为     | 真实调用外部系统               |
| **维护方**  | 团队内部 / 社区           | 工具提供方（GitHub、Notion 等） |
| **迁移成本** | 极低（纯 Markdown，开放标准） | 中等（需要 Server 实现和运维）    |
| **安全边界** | 提示层（不触碰实际系统）        | 执行层（直接操作外部系统）          |


---

### Skills 的优势与局限

**优势：**

- **零基础设施成本**：一个 Markdown 文件就是一个 Skill，不需要部署任何服务
- **团队知识沉淀**：把最资深员工的方法论固化为可复用资产，新人第一天就能产出老员工水准的成果
- **跨工具可移植**：SKILL.md 是开放标准，在 Claude Code、Cursor、OpenClaw、Hermes 等 16+ 工具里直接通用
- **安全可控**：Skill 只影响 AI 的"思路"，不直接操作任何外部系统

**局限：**

- **无法真正"动手"**：Skill 告诉 AI 怎么发邮件，但真的发出去需要 Tool/MCP
- **依赖 description 精准度**：description 写得模糊，Skill 无法被正确命中，等于白写
- **执行质量依赖模型能力**：Skill 是指导，最终质量还取决于 LLM 本身的水平

---

### MCP 的优势与局限

**优势：**

- **真实执行能力**：AI 从"动嘴"变"动手"，可以真正创建 Issue、发送 Slack 消息、查询数据库
- **标准化生态红利**：接入一次 MCP Server，所有支持 MCP 的 AI 工具直接复用
- **实时数据访问**：通过 Resources 直接读取最新文件内容、数据库状态，而不是凭训练记忆

**局限：**

- **运维成本**：MCP Server 是生产级服务，需要部署、监控、持续维护
- **安全风险更高**：直接操作外部系统，权限设置不当可能造成破坏性操作
- **工具清单占用上下文**：接入太多 MCP Server，工具描述会大量占用上下文窗口，分散模型注意力
- **第三方 Server 信任问题**：恶意 Server 可能在 description 里嵌入隐藏指令（工具投毒攻击）

### 实际工作场景对比（经验分享）

#### 场景 1：产品经理写 PRD

**用 Skill 就够了** ✅

```
触发：用户说"帮我写一份登录模块的 PRD"
加载：prd-writing Skill
执行：AI 按照团队的 PRD 模板——背景、用户故事、功能详述、
      验收标准、边界条件——逐步输出，格式和风格与团队规范一致
```

这个场景不需要访问任何外部系统，只需要 AI 按照固定格式方法论工作。**Skill 完全胜任，引入 MCP 反而过度设计。**

---

#### 场景 2：工程师日常 Bug 处理

**Skill + MCP 协同** ✅

```
用户：线上 API 响应变慢了，帮我排查

[Skill: incident-investigation 触发，指导排查步骤]
Step 1: [MCP: CloudWatch] 查看是否有活跃告警
Step 2: [MCP: AWS Logs]   获取最近 30 分钟错误日志
Step 3: [MCP: GitHub]     搜索最近 24 小时相关提交
→ 发现：某次提交引入了 N+1 查询问题

[MCP: GitHub]  创建 Issue，描述问题和复现步骤
[MCP: Slack]   通知 #backend 频道
```

Skill 负责"按什么步骤排查"，MCP 负责"真的去查、真的去通知"。两者缺一不可。

---

#### 场景 3：销售团队每日报告生成

**MCP 为主，Skill 辅助** ✅

```
定时任务（每日 18:00）：

[MCP: Salesforce]  拉取今日成交记录和漏斗数据
[MCP: Google Sheets] 写入数据看板
[Skill: sales-report] 按销售报告规范生成文字摘要
[MCP: Slack] 发送到 #sales-daily 频道
```

这个场景的核心价值在于"真实获取数据、真实写入系统、真实发送通知"，MCP 是主角。Skill 负责数据解读的方法论。

---

#### 场景 4：新人代码 Review 培训

**只用 Skill** ✅

```
触发：新入职工程师提交代码，请求 Review
加载：code-review Skill

[严重] 第 47 行：SQL 拼接存在注入风险 → 建议使用参数化查询
[建议] 第 83 行：循环内数据库查询，O(n) 复杂度 → 建议批量查询
[可选] 第 102 行：变量命名不清晰 → 建议改为 userBillingRecord
```

Skill 把资深工程师的 Review 标准"克隆"给每一次 AI 辅助的代码审查，不需要接任何外部系统。

---

#### 场景 5：跨系统的新员工入职流程

**纯 MCP，多工具编排** ✅

```
触发：HR 系统新员工记录创建

[MCP: GitHub]     邀请加入组织，添加对应仓库权限
[MCP: Notion]     从模板创建新员工 Onboarding 页面
[MCP: Slack]      加入对应频道组，发送欢迎消息
[MCP: Google Calendar] 创建第一周的 1:1 日程
[MCP: Jira]       创建 Onboarding 任务看板

→ Human-in-the-Loop：所有操作汇总给 HR 确认后执行
```

这是纯执行类场景，核心价值是"真实操作多个系统、替人完成繁琐的跨平台手工操作"。

---

#### 场景 6：Redmine 需求-任务-分支闭环开发流程

**Skill + MCP + Git 自动化协同** ✅

```
前置配置（Skill）：
  redmine_base_url
  redmine_api_key

输入：需求 issue 号（例如 #1280）
  [Skill] 识别进入“需求级开发流”
  [MCP: Git] 创建需求分支：
    feature/作者名/1280

输入：任务 issue 号（例如 #1326）
  [Skill] 识别需求-任务关联关系，进入“任务级执行流”
  [MCP: Git] 创建任务分支：
    feature/作者名/1280/1326
  [MCP: Redmine] 将任务状态更新为“开发中”

任务计划与生成：
  [MCP: Redmine] 拉取需求下开发/测试任务描述
  [Skill] 将任务描述结构化为执行计划提示词（范围、约束、验收标准）
  [Skill+LLM] 按计划生成代码
  → Human-in-the-Loop：人工确认代码正确后再继续

finish 指令：
  [MCP: Git] 将任务分支合并回需求分支
  [MCP: Redmine] 将任务状态更新为“已完成”
  [MCP: Redmine] 登记任务工时
```

这个场景里，Skill 负责流程规范、提示词构建与步骤编排，MCP 负责 Redmine/Git 的真实读写与状态变更，属于典型的"方法论 + 执行能力"协同场景。

---

#### 场景 7：完整需求研发流程（Specs 驱动）

**Skill + MCP + Workflow Engine + Human-in-the-Loop 协同** ✅

```
业务背景：
  - 需求文档：钉钉
  - 代码仓库：GitLab
  - 测试用例：自研测试工作台
  - CI/CD：自研应用交付平台
  - 项目任务：Redmine（需求/开发/测试任务）
  - 研发方法：Specs 驱动（先规格、后实现、再验证）
```

**阶段化设计（做什么用 Skill，怎么做用 MCP，何时需要其他机制）**


| 阶段          | 目标                 | Skill 职责（方法论）                         | MCP 职责（真实执行）                                      | 其他关键机制                                |
| ----------- | ------------------ | ------------------------------------- | ------------------------------------------------- | ------------------------------------- |
| 1. 需求摄入与澄清  | 从钉钉需求形成统一问题定义      | 需求澄清清单、冲突检测规则、验收口径模板                  | 读取钉钉需求正文/附件，抓取 Redmine 需求任务元信息                    | Human-in-the-Loop：产品经理确认范围与优先级        |
| 2. Specs 建模 | 产出可执行规格（业务规格+技术规格） | Spec 写作规范（目标、约束、接口、验收标准、非功能要求）        | 将 Spec 写入 GitLab 仓库（如 `specs/*.md`），回写 Redmine 链接 | 版本化规范：Spec 与代码同仓管理，支持评审               |
| 3. 任务分解与排期  | 将需求拆为开发/测试任务       | WBS 拆分策略、依赖关系规则、Definition of Done 模板 | 在 Redmine 创建/更新开发任务与测试任务，维护父子关系                   | 调度器：按优先级和依赖自动排期建议                     |
| 4. 开发实施     | 按 Spec 生成并迭代代码     | 实现步骤模板、代码风格规则、风险检查清单                  | 从 GitLab 创建分支/MR，读取历史提交与关联 MR                     | Branch Policy：命名规范、保护分支、评审门禁          |
| 5. 测试设计与执行  | 基于 Spec 构建可追溯测试闭环  | 用例设计框架（正常/边界/异常/回归）与追踪矩阵模板            | 在测试工作台创建/同步测试用例与执行结果                              | Traceability Matrix：Spec 条款 ↔ 用例 ↔ 缺陷 |
| 6. 持续交付与发布  | 通过流水线完成可观测发布       | 发布检查清单、回滚预案模板、发布说明模板                  | 调用交付平台触发 CI/CD、查询流水线状态、发布结果回写                     | Release Gate：测试通过率/阻塞缺陷阈值控制           |
| 7. 收尾与沉淀    | 形成可复用组织资产          | 复盘模板、最佳实践提炼、失败案例归档规则                  | Redmine 任务状态流转、工时登记、归档链接更新                        | 经验库：沉淀为新 Skill/Prompt 模板，持续优化         |


**一条端到端执行链（示意）**

```
Step 1  [MCP: 钉钉] 读取需求文档
Step 2  [Skill: spec-authoring] 生成 Spec 初稿（目标/约束/接口/验收）
Step 3  [MCP: GitLab] 提交 specs 文档并发起评审
Step 4  [MCP: Redmine] 创建开发/测试子任务并绑定 Spec
Step 5  [Skill: implementation] 基于 Spec 生成实现计划与代码变更
Step 6  [MCP: GitLab] 创建 MR，触发代码评审与静态检查
Step 7  [MCP: 测试工作台] 生成并执行测试用例，回传结果
Step 8  [MCP: 交付平台] 触发 CI/CD，满足门禁后发布
Step 9  [MCP: Redmine] 完成状态流转、登记工时、关闭任务
Step 10 [Skill: retrospective] 自动生成复盘并沉淀可复用策略
```

这个场景的核心不是“单点工具调用”，而是“多系统流程编排”。Skill 保证每一阶段按正确方法执行，MCP 保证每一步都能真实落地到外部系统；而 Workflow Engine、门禁策略与人工确认负责把风险控制在可接受范围内。

---

**选择原则总结：**


| 判断问题                   | 答案  | 选择               |
| ---------------------- | --- | ---------------- |
| 这个任务有没有"讲究"的方法论？       | 有   | Skill            |
| 需不需要访问外部系统的实时数据？       | 需要  | MCP Resources    |
| 需不需要真实执行某个操作（发消息/写文件）？ | 需要  | MCP Tools        |
| 同一类任务会不会重复发生？          | 会   | Skill（固化方法论）     |
| 需不需要跨多个平台协同？           | 需要  | MCP（多 Server 编排） |

## 三、Hermes Agent：Skills 与 MCP 工作机制深度解析

Hermes 是 Nous Research 于 2026 年 2 月发布的开源 Agent 框架，其核心差异化是**"闭环学习"**——Agent 会自动从任务经验中提炼 Skill，越用越强。

### 3.1 整体架构

```
Layer 1 — 接入层（Hermes Gateway）
  15+ 消息平台 → 统一接入 → 跨平台保持持久状态

Layer 2 — 核心 Agent 层（AIAgent）
  Context Engine → Model Router → Tool Registry → Auxiliary LLM
  Skill Engine → Memory System → Scheduler

Layer 3 — 执行后端层（可插拔 Backends）
  Terminal（6种）/ Browser（5种）/ Web（4种）/ File / Vision / MCP
```

**关键设计决策**：执行层与智能层物理分离——Agent 的智能运行在一台机器，代码执行发生在另一台隔离容器，支持高安全性部署场景（如连接远程 HPC 集群而不直接暴露）。

---

### 3.2 Hermes 的 Skill 工作机制：五步闭环学习飞轮

这是 Hermes 最核心的差异化设计，其他框架（包括 OpenClaw）都没有实现：

```
① 执行（Execute）
   Agent 完成一个复杂任务

② 评估（Evaluate）
   自动检测：本次执行是否涉及 ≥5 次工具调用？
   若是，触发 Skill 提炼流程

③ 提取（Extract）
   将本次推理过程提炼为结构化 Skill 文档：
   - 适用场景描述
   - 标准步骤序列
   - 已知陷阱和边界条件
   - 验证步骤

④ 精化（Refine）
   后续遇到相似任务，对比新结果与已有 Skill
   若有矛盾或改进，自动 patch Skill 文档

⑤ 复用（Reuse）
   下次遇到相似任务，直接从 Skill 库检索应用
```

**实际效果**：Nous Research 发布的基准数据显示，使用自创 Skill 的 Agent 完成研究类任务比全新实例**快 40%**。代价是额外消耗约 **15%～25%** 的 Token。

---

### 3.3 Hermes 的 Skill 加载机制：渐进式披露

Hermes 遵循 SKILL.md 开放标准，采用三级渐进式加载，避免上下文膨胀：

```
Session 启动时：
  注入精简 Skill 候选列表（名称 + 描述）
  → 仅几百 Token，几乎不占空间

任务匹配时：
  读取目标 Skill 的摘要部分
  → 确认是否真的相关

需要详细执行时：
  展开完整 SKILL.md 内容
  → 步骤、示例、注意事项全部注入
```

---

### 3.4 Hermes 的四层记忆体系（Skill 所处的位置）

理解 Skill 在 Hermes 中的位置，需要先看它的记忆架构：

```
Layer 1 — 冻结系统提示记忆（始终注入，每次 session 都在）
  MEMORY.md（~2200字符上限）：项目环境、技术栈、配置偏好
  USER.md（~1375字符上限）：沟通风格、工作习惯、时区

Layer 2 — 情节记忆（就是 Skills）
  ~/.hermes/skills/ 下自动生成的 Skill 文档
  从任务经验提炼，渐进式加载

Layer 3 — Session 搜索档案
  SQLite FTS5 全文索引，所有历史对话可检索
  Agent 通过 session_search 工具显式调用
  检索延迟约 10ms（万级文档规模）

Layer 4 — 可选扩展
  Honcho（跨会话用户建模）、Mem0 等第三方记忆提供者
```

**关键设计原则**：架构层保证记忆，而非依赖 LLM 的"启发式判断"。

这与 OpenClaw 形成根本差异：

```
OpenClaw：
  用户说"记住这个" → Agent 决定是否写入 → 可能不写
  会话压缩时 → 可能遗漏
  → 记忆丢失是"预期结果"，而非 Bug

Hermes：
  MEMORY.md → 每次 session 始终注入（无需 Agent 决策）
  Skill → 5次+工具调用后自动触发生成
  FTS5 索引 → 显式调用保证可达性
  → 记忆保证性更高，代价是更高 Token 成本
```

---

### 3.5 Hermes 的 MCP 工作机制：动态接入执行后端

Hermes 的 MCP 集成位于 **Layer 3 执行后端层**，与智能层物理分离。

**工具体系规模**：

- 47 个内置工具
- 19 个预设工具集（toolsets）
- 支持任意 MCP Server 动态接入

**执行后端矩阵**：


| 后端类型       | 种数  | 典型实现                                       |
| ---------- | --- | ------------------------------------------ |
| Terminal   | 6种  | 本地、SSH、Docker、Singularity/Apptainer、远程 GPU |
| Browser    | 5种  | Browser Use 集成                             |
| Web/Search | 4种  | Brave、Tavily 等                             |
| MCP        | 动态  | 任意 MCP Server                              |


**MCP 在 Hermes 中的完整调用链**：

```
用户消息到达
       ↓
AIAgent（run_agent.py）加载系统提示
  = SOUL.md（人格）
  + MEMORY.md（长期记忆）
  + USER.md（用户画像）
  + Skills 候选列表
  + 工具指南（含 MCP Tools 描述）
       ↓
API 调用（主 LLM 推理）
       ↓
检测到工具调用指令？
       ├── 是 → Tool Registry 路由 → 对应执行后端
       │         本地命令 → Terminal Backend
       │         外部 API → MCP Server
       │         网页操作 → Browser Backend
       │         执行结果注入上下文 → 循环
       └── 否 → 生成最终响应 → 持久化到 SQLite → 结束
```

**辅助 LLM（Auxiliary LLM）**：Hermes 额外维护一个与主模型独立的辅助 LLM，专门处理视觉分析、摘要等辅助任务，避免这些任务干扰主对话流。

### 3.6 Skill 与 MCP 在 Hermes 中的协作示意

一个典型的"代码库分析 + 自动创建文档"任务，展示两者如何协作：

```
用户：帮我分析这个 Rust 项目的架构，生成一份 ADR 文档

[Skill 加载]
  architecture-analysis Skill 命中
  → 注入：先读目录结构、再读核心模块、
          最后按 ADR 模板（背景/决策/后果）输出

[MCP 执行]
  [MCP: filesystem] 读取项目目录树
  [MCP: filesystem] 读取 Cargo.toml、src/main.rs、src/lib.rs
  [MCP: github]     查看最近 PR，了解最新变更意图

[Skill 指导]
  按 ADR Skill 定义的格式组织分析结果

[MCP 执行]
  [MCP: notion]     创建 ADR 页面，写入分析内容
  [MCP: slack]      通知团队："架构文档已更新"

[Skill 自动提炼]
  本次涉及 8 次工具调用（>5 阈值）
  → 自动生成 "rust-project-analysis" Skill
  → 下次同类任务直接复用，执行效率提升
```

### 小结：Hermes 设计的核心哲学

Hermes 本质上是在回答一个问题：**"如何让 Agent 从每次工作中真正学到东西，而不是每次都从零开始？"**

- **Skill** 是它的"程序性记忆"——从任务经验自动提炼，越用越精准
- **MCP** 是它的"执行双手"——物理分离保证安全，动态接入保证灵活
- **记忆分层**——架构层保证关键信息一定在，而不是寄希望于 LLM 自己"记住"

这三者在 Hermes 中不是孤立模块，而是形成了一个真正的**闭环学习系统**：工具执行产生经验 → 经验提炼为 Skill → Skill 加速下次工具执行 → 继续积累更多经验。
